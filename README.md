# Magic Roguelike

Монорепозиторий с **авторитетной симуляцией на сервере** и **React/Three‑клиентом**: хаб, забег, защита механизма, волны врагов, скиллы с рунной схемой, снапшот состояния по WebSocket.

## Стек

| Слой | Пакет | Технологии |
|------|--------|------------|
| Симуляция и протокол | `shared/` | TypeScript, чистый `stepSim`, типы команд и `GameSnapshot` |
| Сервер | `server/` | Node.js, TypeScript, `ws`, комната и тик |
| Клиент | `client/` | Vite, React 19, TypeScript, Zustand, Three.js |
| Ассеты иконок | `assets/` | пайплайн PNG → атлас (скрипты в корневом `package.json`) |

**Корень репозитория:** npm workspaces (`shared`, `server`, `client`).

## Быстрый старт

```bash
npm install
npm run dev
```

- Клиент обычно на **http://localhost:5173**, WebSocket — как настроено (см. `client/.env.local` и переменную `VITE_WS_URL` при необходимости).
- Сборка всего стека: `npm run build`.

## Идея и направление

Игра про мага и **артефакт реальности**: из хаба запускается **забег** — таймер фазы артефакта, волны с порталов, защита **механизма** и героя, руны **Q/W/E**, **Invoke** и каст по прицелу на земле. Мета‑прогресс и кооп расширяют цикл позже; ядро — **читаемый бой** и баланс на общей симуляции в `shared/`.

Детали интерфейса, пайплайна иконок и правил арта — в **`client/README.md`**, **`assets/README.md`** и **`.cursor/rules/`**.

## Как работаем (договорённости)

- **Ты** задаёшь **направление, приоритеты и геймдизайн/продукт**; решения «что строим дальше» остаются за тобой.
- **Вся реализация в репозитории** (код, конфиги, правки, сборка, типичная отладка) — **на стороне агента**; сценарий «ты сам пишешь код по инструкции» не считается нормой.
- **Спорные технические выборы** (библиотека, расклад папок, граница client/shared) агент прогоняет через **практику из правил репо** и здравый смысл; продуктовые вещи (как должно ощущаться в бою) лучше уточнять у тебя.
- **Планы задач** ведём по циклу: **сформулировали фокус → сделали в коде → закрыли**; не накапливаем долгоживущие «план‑файлы» как обязательный артефакт (актуальность — в коде, коммитах и при необходимости в Issues/описании PR).

## Планы в Cursor

В проекте намеренно **нет закреплённого markdown‑плана в git**: временные планы можно составлять в Cursor Plan Mode или на бумаге у себя, а после выполнения — удалять или не коммитить. Папка **`.cursor/plans/`** зарезервирована только чтобы не плодить устаревшие документы; правила — в **`.cursor/rules/`**.

Файлы планов **вне репозитория** (личный каталог Cursor) в git **не попадают** — их тоже можно очистить у себя по тому же принципу.
